### 1. Public Preview: Azure Backup for Elastic SAN 

**公開日時**: 2026年04月24日 18:30:50 UTC
**リンク**: [Public Preview: Azure Backup for Elastic SAN ](https://azure.microsoft.com/updates?id=560904)

**アップデートID**: 560904
**情報源**: Azure Updates API

**カテゴリ**: In preview, Storage, Azure Elastic SAN, Features

**要約**:

- 何が更新されたか  
Azure BackupがElastic SANをサポートし、Elastic SANボリュームのバックアップおよびリストア機能がパブリックプレビューとして提供開始されました。

- 主な変更点や新機能  
Elastic SANのデータをAzure Backupでフルマネージドにバックアップ・リストアできるようになりました。これにより、誤削除やランサムウェア攻撃、アプリケーションアップデートなどによるデータ損失からElastic SANボリュームを保護できます。

本機能はパブリックプレビュー段階のため、本番環境での利用には注意が必要です。また、サポート範囲や制限事項については公式ドキュメントを確認してください。

**詳細**:

本アップデートは、「Azure Backup for Elastic SAN」のパブリックプレビュー提供開始に関するものです。これにより、Azure BackupがElastic SAN（Elastic Storage Area Network）をサポートし、Elastic SANボリュームのバックアップおよびリストアをフルマネージドで実行できるようになりました。アップデートの目的は、Elastic SAN上のデータを偶発的な削除、ランサムウェア攻撃、アプリケーションのアップデートなどによるデータ損失から保護することです。

具体的な機能としては、Azure BackupとElastic SANの統合により、Elastic SANボリュームのバックアップデータをエクスポートし、必要に応じてリストアできる点が挙げられます。これにより、従来は手動で行っていたバックアップ・リストア作業をAzure Backupの管理機能を通じて自動化・効率化することが可能となります。バックアップはフルマネージドで提供されるため、インフラ管理者はバックアップのスケジューリングや保持ポリシーの設定、リストアポイントの管理などをAzureポータルやAPIを通じて一元的に実施できます。

技術的な仕組みとしては、Azure BackupサービスがElastic SANのボリュームデータをエクスポートし、バックアップストレージに安全に保存します。リストア時には、バックアップデータから指定したボリュームを復元することができます。これにより、データの整合性や可用性を確保しつつ、迅速な障害対応やデータリカバリが実現できます。

活用シナリオとしては、ミッションクリティカルなデータをElastic SAN上で運用している場合の定期バックアップ、アプリケーションのバージョンアップ前の事前バックアップ、誤操作によるデータ削除からの迅速な復旧、ランサムウェア等のセキュリティインシデント発生時のリカバリなどが想定されます。これにより、企業のBCP（事業継続計画）やDR（ディザスタリカバリ）対策の強化にも寄与します。

注意点や制限事項については、本アップデートがパブリックプレビュー段階であるため、本番環境での利用は慎重に検討する必要があります。また、Elastic SANのバックアップ対象やリストアの粒度、サポートされるリージョンやバックアップ保持期間など、詳細な制限事項については公式ドキュメントを参照することが推奨されます。

関連するAzureサービスとの連携としては、Azure Backupの他のストレージサービス（例：Azure Disk BackupやAzure Files Backup）と同様に、統合的なバックアップ管理が可能です。これにより、複数のストレージサービスを横断したデータ保護戦略をAzure上で一元的に実装することができます。

技術的な仕組みとしては、Azureが提供する分離型の実行基盤上でエージェントがデプロイされ、各セッションごとに新規のサンドボックス環境がプロビジョニングされます。この環境はセッション終了時に破棄されるため、永続的なデータや状態が残ることはありません。これにより、セキュアなマルチテナント運用や、機密性の高いワークロードの実行が可能となります。

活用シナリオとしては、CI/CDパイプラインのビルド・デプロイ作業や、セキュリティ要件が厳しい業務プロセスの自動化、複数のプロジェクトやチームでのエージェント共有運用などが挙げられます。特に、セッションごとに完全な分離が求められる環境や、エージェント実行時のクリーンな状態が必要なケースで有効です。

注意点としては、パブリックプレビュー段階であるため、サービスの安定性や機能面で一部制限が存在する可能性があります。また、既存のFoundry Agent Serviceとの互換性や、サンドボックス環境におけるリソース制限などについては、公式ドキュメントやサポート情報を参照する必要があります。

関連するAzureサービスとの連携については、Foundry Agent Service自体がAzure上で動作するため、Azure DevOpsやAzure Pipelinesなどの自動化・CI/CDサービスとの統合が想定されます。これにより、Azureエコシステム内でセキュアかつ効率的なエージェント運用が実現できます。

### 4. Generally Available: WireGuard In‑Transit Encryption for Azure Kubernetes Service (AKS)

**公開日時**: 2026年04月24日 18:15:17 UTC
**リンク**: [Generally Available: WireGuard In‑Transit Encryption for Azure Kubernetes Service (AKS)](https://azure.microsoft.com/updates?id=560015)

**アップデートID**: 560015
**情報源**: Azure Updates API

**カテゴリ**: Launched, Compute, Containers, Azure Kubernetes Service (AKS), Features, Security

**要約**:

- 何が更新されたか  
Azure Kubernetes Service（AKS）において、WireGuardによるインフライト（転送中）暗号化機能が一般提供（GA）されました。

- 主な変更点や新機能  
Azure CNI（Ciliumベース）およびAdvanced Container Networking Services（ACNS）を利用しているAKSクラスターで、WireGuardを用いたノード間通信の自動暗号化が可能になりました。これにより、クラスターノード間のトラフィックを透過的に暗号化し、セキュリティを強化できます。

- 影響を受ける対象  
Azure CNI（Cilium）およびACNSを利用しているAKSクラスターが対象です。既存または新規の該当構成のクラスターで利用できます。

- 注意点があれば記載  
WireGuard暗号化を利用するには、AKSクラスターがAzure CNI（Cilium）またはACNSを使用している必要があります。その他のネットワークプラグインや構成では利用できません。導入前に、既存のネットワーク設定やパフォーマンスへの影響を十分に確認することを推奨します。

**詳細**:

本アップデートは、Azure Kubernetes Service（AKS）において、WireGuardによるインフライト暗号化が一般提供（GA）となったことを示しています。対象となるのは、Azure CNI（Container Networking Interface）をCiliumおよびAdvanced Container Networking Services（ACNS）と組み合わせて利用しているAKSクラスターです。この機能の導入背景には、Kubernetesクラスター内のノード間通信におけるセキュリティ強化の要請があります。従来、AKSのネットワーク通信は暗号化されていない状態で転送されることが多く、特にマルチテナント環境や厳格なコンプライアンス要件を持つシナリオにおいては、通信経路の暗号化が求められていました。

WireGuardは、高速かつシンプルなVPNプロトコルであり、Linuxカーネルにネイティブ統合されている点が特徴です。今回のアップデートにより、AKSクラスター内のノード間通信がWireGuardを用いて自動的かつ透過的に暗号化されるようになります。これにより、アプリケーションやワークロードの変更を伴うことなく、ネットワークレベルでのセキュリティ向上を実現できます。具体的には、Azure CNIをCiliumおよびACNSと併用している環境でWireGuardが有効化されることで、ノード間のパケットがWireGuardトンネルを通じて暗号化され、外部からの盗聴や改ざんリスクを低減します。

技術的な実装としては、Ciliumが提供するネットワークポリシーおよびデータプレーン機能と、WireGuardのカーネルモジュールを組み合わせることで、ノード間の通信を自動的に暗号化します。これにより、Kubernetesクラスターの管理者は、追加の構成や複雑な証明書管理を行うことなく、セキュアな通信経路を確保できます。

活用シナリオとしては、機密性の高いデータを扱う業務システムや、複数のテナントが共存するマルチテナント環境、あるいは厳格なセキュリティ基準を満たす必要がある金融・医療分野のシステムなどが挙げられます。これらのシナリオでは、ノード間通信の暗号化が必須要件となる場合が多く、本アップデートによりAzure上での要件充足が容易になります。

注意点としては、WireGuardによる暗号化はAzure CNIをCiliumおよびACNSと組み合わせて利用しているAKSクラスターに限定される点です。また、既存のネットワーク構成やパフォーマンス要件に影響を与える可能性があるため、導入前には十分な検証が推奨されます。

本機能は、Azure Kubernetes Serviceのセキュリティ強化に寄与し、Azure CNI、Cilium、ACNSといった関連サービスとの連携によって、より堅牢なクラウドネイティブ環境の構築を支援します。

技術的な仕組みとして、Azure App Serviceは複数の.NETバージョンをサポートしており、アプリケーションの設定やデプロイ時に使用する.NETバージョンを指定できます。アップグレードの際は、アプリケーションのコードや依存ライブラリが.NET 10（LTS）に対応しているかを事前に確認し、必要に応じてリファクタリングやテストを行うことが重要です。App Serviceの構成変更やCI/CDパイプラインの更新も必要になる場合があります。

活用シナリオとしては、企業の業務システムやWeb API、SaaSアプリケーションなどがApp Service上で.NET 8（LTS）を利用している場合、サポート終了前に.NET 10（LTS）への移行を計画し、段階的なテストとリリースを実施することが推奨されます。これにより、セキュリティやパフォーマンスの向上、最新機能の活用が可能になります。

注意点として、サポート終了後もアプリケーションが動作するため、移行作業を怠りがちですが、セキュリティリスクや障害対応の負担増加を避けるため、計画的なアップグレードが必須です。また、移行時には依存ライブラリや外部サービスとの互換性も十分に検証する必要があります。
