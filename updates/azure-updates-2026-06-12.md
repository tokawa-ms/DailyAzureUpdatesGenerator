### 1. Retirement: Av2-series, F-series, Fs-series, Fsv2-series, G-series, Gs-series, and Lsv2-series Virtual Machines for Azure Batch pools

**公開日時**: 2026年06月11日 19:00:41 UTC
**リンク**: [Retirement: Av2-series, F-series, Fs-series, Fsv2-series, G-series, Gs-series, and Lsv2-series Virtual Machines for Azure Batch pools](https://azure.microsoft.com/updates?id=564774)

**アップデートID**: 564774
**情報源**: Azure Updates API

**カテゴリ**: Compute, Batch, Retirements

**要約**:

- 何が更新されたか  
Azure Computeは、Azure Batchプールで利用されている以下の仮想マシン（VM）シリーズのサポートを2028年11月15日に終了します。

- 主な変更点や新機能  
Av2-series、F-series、Fs-series、Fsv2-series、G-series、Gs-series、Lsv2-seriesの各VMシリーズが廃止対象となります。廃止日以降、これらのVMシリーズを利用した新規Batchプールの作成や既存プールのスケールアウトができなくなります。

- 影響を受ける対象  
現在Azure Batchで上記VMシリーズを利用しているユーザーやシステムが対象です。既存のBatchプールやワークロードでこれらのVMシリーズを使用している場合、移行が必要となります。

- 注意点  
2028年11月15日以降は、これらのVMシリーズを利用したBatchプールの新規作成や拡張ができなくなります。既存のワークロードへの影響を避けるため、早めにサポート対象のVMシリーズへの移行計画を検討してください。詳細は公式ドキュメントをご確認ください。

**詳細**:

2028年11月15日をもって、Azure ComputeはAv2シリーズ、Fシリーズ、Fsシリーズ、Fsv2シリーズ、Gシリーズ、Gsシリーズ、Lsv2シリーズの仮想マシン（VM）をAzure Batchプールでの利用対象からリタイアします。このアップデートは、これらのVMシリーズに依存しているAzure Batchプールに直接影響を及ぼします。リタイア後は、これらのVMシリーズを新規に利用したBatchプールの作成や既存プールへの追加ができなくなります。

今回の変更の背景には、Azureプラットフォームの進化と、より新しい世代の仮想マシンへの移行促進があります。これにより、パフォーマンスやセキュリティ、コスト効率の向上が期待されます。対象となるVMシリーズは、いずれも比較的古い世代に属しており、今後はより新しいSKUへの移行が求められます。

技術的には、Azure Batchは大規模な並列処理やバッチジョブの実行に最適化されたサービスであり、仮想マシンのプールを動的に管理します。今回リタイア対象となるVMシリーズは、これまで多様なバッチワークロードで利用されてきましたが、今後はサポートされなくなるため、Batchプールの構成時にこれらのSKUを指定することができなくなります。既存のプールについても、リタイア日以降はスケールアウトやノード追加時にエラーとなる可能性があるため、事前の移行が必須となります。

活用シナリオとしては、これらのVMシリーズは主に科学技術計算、メディア処理、金融リスク分析など、計算リソースを大量に必要とするバッチ処理で利用されてきました。今後は、より新しいDシリーズやEシリーズなど、現行サポートされているVMシリーズへの移行が推奨されます。

注意点として、リタイア日以降は新規プール作成や既存プールのスケール操作が制限されるため、十分な移行計画が必要です。また、これらのVMシリーズに依存したカスタムイメージやジョブスケジューリングの仕組みを利用している場合、互換性の確認や再構成が求められます。

Azure Batchは、Azure StorageやAzure Key Vaultなど他のAzureサービスと連携してジョブデータの管理やセキュリティを強化できますが、VMシリーズのリタイアによってこれらの連携機能自体に影響はありません。ただし、基盤となるVMの選定には最新のサポート状況を考慮する必要があります。詳細は公式アップデートページ（https://azure.microsoft.com/updates?id=564774）を参照してください。

### 2. Retirement: D-series, Ds-series, Dv2-series, Dsv2-series, and Ls-series Virtual Machines for Azure Batch pools

**公開日時**: 2026年06月11日 19:00:41 UTC
**リンク**: [Retirement: D-series, Ds-series, Dv2-series, Dsv2-series, and Ls-series Virtual Machines for Azure Batch pools](https://azure.microsoft.com/updates?id=564772)

**アップデートID**: 564772
**情報源**: Azure Updates API

**カテゴリ**: Compute, Batch, Retirements

**要約**:

- 何が更新されたか  
Azure Computeは、Azure Batchプールで利用されている以下の仮想マシン（VM）シリーズのサポートを2028年5月1日に終了します。対象はD-series、Ds-series、Dv2-series、Dsv2-series、Ls-seriesです。

- 主な変更点や新機能  
2028年5月1日以降、上記VMシリーズを利用した新規のBatchプールは作成できなくなります。既存のBatchプールも、これらのVMシリーズを引き続き利用することはできなくなります。

- 影響を受ける対象  
現在、D-series、Ds-series、Dv2-series、Dsv2-series、Ls-seriesのいずれかを利用しているAzure Batchプールが対象です。これらのVMシリーズを利用しているワークロードやジョブは、今後の移行が必要となります。

- 注意点  
対象VMシリーズを利用している場合は、2028年5月1日までに推奨される後継VMシリーズへの移行計画を立てる必要があります。移行しない場合、Batchプールの新規作成や運用継続ができなくなるため、早めの対応が推奨されます。

詳細: [Azure Update公式情報](https://azure.microsoft.com/updates?id=564772)

**詳細**:

Azure Computeは、2028年5月1日をもってD-series、Ds-series、Dv2-series、Dsv2-series、Ls-seriesの仮想マシン（VM）をAzure Batchプールで利用する機能を廃止することを発表しました。本アップデートの背景には、これらのVMシリーズが旧世代であり、Azureプラットフォーム全体のパフォーマンスやセキュリティ、運用効率の向上を目的としたサービスの刷新が挙げられます。Azureは定期的にVMファミリーのラインナップを見直し、より高性能かつコスト効率の良い新世代VMへの移行を推進しています。

具体的な変更内容としては、2028年5月1日以降、Azure Batchプールにおいて上記のVMシリーズを新規に作成することができなくなります。既存のBatchプールでこれらのVMシリーズを利用している場合、アップデート後も新規作成は不可となりますが、既存プールの運用に関しては詳細が明記されていません。Azure Batchは大規模な並列処理やバッチジョブの管理を行うサービスであり、VMシリーズの選択はジョブのパフォーマンスやコストに直結します。

技術的な仕組みとして、Azure Batchはプール単位でVMを管理し、ユーザーはプール作成時にVMシリーズを指定します。今回のアップデートにより、プール作成時の選択肢から該当VMシリーズが除外されることになります。既存プールの維持や移行については、今後Azure公式ドキュメントやサポート情報を参照する必要があります。

活用シナリオとして、これらのVMシリーズは従来、一般的な計算処理やストレージ最適化が求められるバッチジョブに多く利用されてきました。例えば、D-seriesやDv2-seriesはバランスの取れたCPU・メモリ構成で幅広い用途に対応し、Ls-seriesはローカルストレージ性能を重視したジョブに適していました。しかし、今後は新世代VM（例：Dv3、Ev4、Lsv2など）への移行が必要となります。

注意点として、2028年5月1日以降は該当VMシリーズを利用した新規Batchプールの作成ができなくなるため、現在これらのVMシリーズを利用しているBatchワークロードについては、事前に新世代VMへの移行計画を立てる必要があります。また、既存プールの運用やジョブ実行に影響が出る可能性があるため、Azure公式の移行ガイドやサポート情報を確認し、適切な対応を行うことが重要です。

関連するAzureサービスとの連携については、Azure BatchはAzure StorageやAzure Active Directory、Azure Monitorなどと密接に連携しており、VMシリーズの変更によってこれらのサービスとの連携機能自体に影響はありません。ただし、VMの性能や特性が変わることでジョブの実行時間やコスト、ストレージI/Oなどのパラメータに変化が生じる可能性があるため、移行後のパフォーマンス検証が推奨されます。

詳細は公式アップデートページ（https://azure.microsoft.com/updates?id=564772）をご参照ください。

### 3. Retirement: GPv1 and Legacy Blob storage account creation

**公開日時**: 2026年06月11日 19:00:41 UTC
**リンク**: [Retirement: GPv1 and Legacy Blob storage account creation](https://azure.microsoft.com/updates?id=564441)

**アップデートID**: 564441
**情報源**: Azure Updates API

**カテゴリ**: Storage, Storage Accounts, Retirements

**要約**:

- 何が更新されたか  
Azure Storageのポートフォリオ整理の一環として、General Purpose v1（GPv1）ストレージアカウントおよびレガシーBlobストレージアカウントの新規作成が廃止されることが発表されました。

- 主な変更点や新機能  
今後、GPv1およびレガシーBlobストレージアカウントの新規作成ができなくなります。これにより、パフォーマンス、スケーラビリティ、コスト効率の向上が期待される最新のストレージアカウント（例：GPv2など）への移行が推奨されます。

- 影響を受ける対象  
GPv1ストレージアカウントおよびレガシーBlobストレージアカウントを新規作成していたユーザーやシステムが影響を受けます。既存のGPv1アカウントには即時の影響はありませんが、今後のサポートや機能追加は制限される可能性があります。

- 注意点があれば記載  
既存のGPv1およびレガシーBlobストレージアカウントを利用している場合は、早めにGPv2などの最新アカウントタイプへの移行を検討してください。新規プロジェクトやリソース作成時は、GPv2ストレージアカウントの利用が必須となります。

**詳細**:

Azureは、ストレージサービスのポートフォリオを合理化し、パフォーマンス、スケーラビリティ、コスト効率を強化するための取り組みの一環として、General purpose v1（GPv1）ストレージアカウントおよびレガシーBlobストレージアカウントの新規作成を廃止することを発表しました。これにより、今後はGPv1ストレージアカウントや従来型のBlobストレージアカウントを新規に作成することができなくなります。

このアップデートの背景には、Azureストレージサービスの進化と、より高機能かつ効率的なストレージアカウントであるGeneral purpose v2（GPv2）や、他の最新ストレージサービスへの移行を促進する目的があります。GPv2アカウントは、GPv1に比べてより優れたパフォーマンスや機能、柔軟な価格体系を提供しており、今後の新規プロジェクトや既存システムの拡張において推奨される選択肢となっています。

技術的には、GPv1アカウントは従来のBlob、Queue、Table、Fileストレージをサポートしていましたが、今後はこれらの用途にもGPv2アカウントの利用が求められます。GPv2アカウントは、階層型ストレージ（ホット、クール、アーカイブ）、高度なセキュリティ機能、より細かなアクセス制御、Azure Data Lake Storage Gen2のサポートなど、多くの拡張機能を備えています。既存のGPv1アカウントは引き続き利用可能ですが、新規作成ができなくなるため、今後の運用設計やシステム構築時にはGPv2アカウントを前提とした設計が必要です。

活用シナリオとしては、従来GPv1アカウントを利用していたアプリケーションやワークロードを、段階的にGPv2アカウントへ移行することが推奨されます。これにより、最新のストレージ機能やコスト最適化、将来的なAzureサービスとの連携強化が期待できます。特に、Azure FunctionsやAzure Logic Apps、Azure Synapse Analyticsなど、ストレージアカウントを利用する他のAzureサービスとの連携においても、GPv2アカウントの利用が標準となります。

注意点として、既存のGPv1アカウントは引き続きサポートされるものの、今後の機能追加や最適化はGPv2アカウントが中心となるため、長期的な運用を考慮した場合は早期の移行計画策定が重要です。また、レガシーBlobストレージアカウントについても同様に新規作成ができなくなるため、Blobストレージの新規利用時にはGPv2アカウントを選択してください。

関連するAzureサービスとの連携においても、GPv2アカウントは最新のAPIやセキュリティ機能、データ分析基盤との統合など、多くの利点があります。今後のAzureストレージ活用においては、GPv2アカウントを中心とした設計・運用が求められます。

詳細は公式アップデート情報（https://azure.microsoft.com/updates?id=564441）をご参照ください。

**詳細**:

SQL MCP Serverが一般提供（GA）となりました。本サービスは、顧客に対して本番データへの制御されたアクセスを実現しつつ、セキュアかつ高パフォーマンスなエージェント型ソリューションの構築手段を提供します。SQL MCP Serverは、SQLベースで設計されており、PostgreSQLおよびAzure Cosmos DBとの互換性を持っています。これにより、既存のSQLワークロードを持つシステムや、オープンソースのPostgreSQL、さらにはNoSQLデータベースであるAzure Cosmos DBを利用している環境でも柔軟に導入することが可能です。

本アップデートの主な目的は、データへのアクセス制御を厳格に行いながら、エージェント型アプリケーションや自動化ソリューションの開発を容易にすることにあります。SQL MCP Serverは、動的なデータアクセス管理機能を備えており、運用環境のセキュリティを損なうことなく、必要なデータへのアクセスを効率的に提供します。これにより、開発者や運用担当者は、データガバナンスを維持しつつ、柔軟なアプリケーション開発や運用自動化を推進できます。

技術的には、SQL MCP ServerはSQLプロトコルをベースとしたアクセスインターフェースを提供し、既存のSQLクライアントやツールとの互換性を確保しています。また、PostgreSQLやAzure Cosmos DBとの互換性により、マルチデータベース環境での統合的な運用が可能です。これにより、異なるデータストア間でのデータアクセスや管理が一元化され、システム全体の運用効率が向上します。

注意点としては、SQL MCP Serverの導入にあたっては、既存のデータベース環境との互換性やアクセス制御ポリシーの設計が重要となります。また、サポートされるデータベースはSQL、PostgreSQL、Azure Cosmos DBに限定されているため、他のデータベース製品との連携には追加の検討が必要です。

関連するAzureサービスとの連携については、Azure Cosmos DBやAzure Database for PostgreSQLなどのPaaS型データベースサービスと組み合わせることで、クラウドネイティブなデータ管理基盤の構築が可能です。SQL MCP Serverは、これらのサービスとの統合運用を支援し、セキュアかつ効率的なデータアクセス管理を実現します。

### 5. Retirement: Azure Load Balancer Inbound NAT rule version 1 for Azure VMSS (aka Inbound NAT Pools)

**公開日時**: 2026年06月11日 16:15:07 UTC
**リンク**: [Retirement: Azure Load Balancer Inbound NAT rule version 1 for Azure VMSS (aka Inbound NAT Pools)](https://azure.microsoft.com/updates?id=565482)

**アップデートID**: 565482
**情報源**: Azure Updates API

**カテゴリ**: Retirements, Services, Feature

**要約**:

【何が更新されたか】  
Azure Load BalancerのInbound NAT rule version 1（特にAzure Virtual Machine Scale Sets向けのInbound NAT Pools）の廃止範囲が変更されました。

【主な変更点や新機能】  
以前は、単一VM向けNATルールとVMSS向けNAT Poolsの両方が廃止対象でしたが、今回の更新で廃止範囲が「VMSS向けのInbound NAT Pools」に限定されました。単一VM向けのNATルールは今回の廃止対象から除外されています。

【影響を受ける対象】  
Azure VMSS（Virtual Machine Scale Sets）でInbound NAT Pools（NAT rule version 1）を利用している環境が影響を受けます。単一VMでNAT rule version 1を利用している場合は今回の廃止対象ではありません。

【注意点】  
VMSSでInbound NAT Poolsを利用している場合は、今後の廃止に備えてNAT rule version 2への移行などの対応が必要です。廃止スケジュールや移行方法については公式ドキュメントを確認し、早めの対応を推奨します。

**詳細**:

本アップデートは、Azure Load BalancerのInbound NAT rule version 1に関するリタイアメント（廃止）についての最新情報です。従来、AzureはInbound NAT rule version 1リソース全体、すなわち単一VM向けのNATルールおよびVirtual Machine Scale Sets（VMSS）向けのNATプールの両方を対象に廃止をアナウンスしていましたが、今回のアップデートにより、その廃止範囲がより限定的になったことが通知されています。

Azure Load BalancerのInbound NAT ruleは、外部からの特定ポートへのトラフィックを、内部の仮想マシンやVMSSのインスタンスの特定ポートに転送するための機能です。これにより、複数のVMやVMSSインスタンスに対して、個別にリモートデスクトップ（RDP）やSSHなどの管理用アクセスを提供することが可能となります。Version 1のNATルールは、これまで長らく利用されてきましたが、Azureのサービス進化に伴い、より新しいバージョンや機能への移行が推奨されています。

本アップデートの技術的なポイントは、VMSS向けのInbound NAT Pools（NATプール）が廃止対象となる点です。NATプールは、複数のインスタンスに対して自動的にポート範囲を割り当て、スケールアウト構成時にも効率的な管理を可能にしていました。廃止後は、これらのバージョン1リソースを新しいバージョンや推奨される構成に移行する必要があります。移行にあたっては、Azure Resource Manager（ARM）ベースのLoad Balancerや、最新のNATルール構成への切り替えが求められます。

利用シナリオとしては、開発環境や運用環境でVMSSを用いて多数の仮想マシンを管理している場合、各インスタンスへの管理アクセスやアプリケーションポートの公開にNATプールが活用されてきました。今後は、これらの構成を見直し、サポートされている最新のNATルールやAzure Load Balancerの機能を利用することが必要です。

注意点として、廃止対象となるリソースは今後サポートやセキュリティアップデートの対象外となるため、運用中のシステムに影響が出る可能性があります。早期に移行計画を策定し、テストや検証を十分に行うことが推奨されます。また、関連するAzureサービス、たとえばAzure Virtual NetworkやNetwork Security Group（NSG）との連携においても、設定変更が必要となる場合があります。移行作業に際しては、既存の構成や依存関係を十分に確認し、Azureの公式ドキュメントやサポート情報を参照することが重要です。
