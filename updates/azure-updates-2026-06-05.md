### 2. Generally Available: Azure Files assessments are now available worldwide using Azure Migrate

**公開日時**: 2026年06月04日 21:45:11 UTC
**リンク**: [Generally Available: Azure Files assessments are now available worldwide using Azure Migrate](https://azure.microsoft.com/updates?id=564563)

**アップデートID**: 564563
**情報源**: Azure Updates API

**カテゴリ**: Launched, Management and governance, Migration, Azure Migrate, Services, Feature

**要約**:

- 何が更新されたか  
Azure Migrateで、SMBおよびNFSファイル共有（WindowsおよびLinuxサーバー上）のディスカバリーとアセスメント機能が一般提供（GA）され、全世界で利用可能になりました。

- 主な変更点や新機能  
これまでAzure Migrateは主に仮想マシンやデータベースの移行評価をサポートしていましたが、新たにファイル共有（SMB/NFS）の環境も自動で検出・評価できるようになりました。これにより、ファイルサーバーの移行計画をデータに基づいて立案できます。

- 影響を受ける対象  
オンプレミスでWindowsまたはLinuxサーバー上にSMB/NFSファイル共有を運用している企業や組織、またはこれらのファイル共有をAzureへ移行検討中の技術者が対象です。

- 注意点があれば記載  
本機能はAzure Migrateの一部として提供されます。利用にはAzure Migrateのセットアップおよび対象サーバーへの必要なエージェントのインストールが必要です。既存のファイル共有環境の構成やアクセス権限設定に注意して評価を実施してください。

**詳細**:

Azure Migrateは、クラウド移行プロジェクトを支援するための統合プラットフォームとして、仮想マシンや物理サーバー、データベースなど多様なワークロードの評価と移行をサポートしています。今回のアップデートにより、Azure MigrateはSMBおよびNFSファイル共有（WindowsおよびLinuxサーバー上でホストされているもの）のディスカバリーとアセスメント機能をグローバルに一般提供開始しました。この機能追加の目的は、企業がオンプレミス環境に存在するファイル共有の現状を正確に把握し、Azure Filesなどクラウドサービスへの移行計画をデータに基づいて立案できるようにすることです。

具体的には、Azure Migrateを利用することで、SMB（Server Message Block）やNFS（Network File System）プロトコルを用いたファイル共有の構成情報や利用状況、容量、アクセスパターンなどを自動的に検出し、評価レポートとしてまとめることが可能です。これにより、移行対象となるファイル共有の特性や移行先のAzure Filesとの適合性、必要なリソースやコスト見積もりなどを詳細に把握できます。

技術的な仕組みとしては、Azure Migrateのアセスメントツールが対象サーバーにエージェントを導入し、ファイル共有のメタデータや利用統計を収集します。収集されたデータはAzure Migrateのポータル上で可視化され、移行計画の策定に活用できます。これにより、従来手作業で行っていたファイル共有の調査や評価プロセスが自動化され、移行プロジェクトの効率化と精度向上が実現します。

活用シナリオとしては、オンプレミスのWindows ServerやLinux Server上に存在するSMB/NFSファイル共有をAzure FilesやAzure NetApp Filesへ移行したい場合に、事前評価としてAzure Migrateを利用することで、移行リスクの低減や最適な移行方法の選定が可能となります。また、複数拠点や大規模なファイル共有環境を持つ企業にとって、全体像を把握した上で段階的な移行計画を立てる際にも有効です。

注意点として、評価対象となるファイル共有がAzure Migrateでサポートされているプロトコル（SMB/NFS）であること、またサーバーへのエージェント導入が必要となる場合がある点に留意する必要があります。さらに、評価結果は移行の意思決定を支援するものであり、実際の移行作業には別途Azure Files Migrationツールやその他の移行サービスとの連携が必要となります。

関連するAzureサービスとしては、Azure FilesやAzure NetApp Files、Azure Storage、Azure Migrateの各評価・移行機能が密接に連携します。これらのサービスを組み合わせることで、オンプレミスからクラウドへのファイル共有移行を包括的に管理・実行することが可能です。

技術的には、SQL拡張による宣言的なパイプライン定義が特徴です。ユーザーはSQL文を用いてAI処理の各ステップを記述し、HorizonDBがその定義に基づいてパイプラインを構築・実行します。これにより、データベースエンジンのスケーラビリティや信頼性を活かしつつ、AI処理をデータベースの近くで効率的に実行できます。

活用シナリオとしては、大規模なテキストデータの取り込みと前処理、AIによる特徴抽出や要約、生成AIを用いたデータ生成、検索ランキングの自動化などが想定されます。たとえば、企業内のドキュメント管理システムにおいて、ドキュメントの自動分類や要約、検索クエリに対する関連度ランキングなどを、データベース内で直接実行することが可能です。

注意点としては、現時点ではパブリックプレビューであるため、運用環境での利用には慎重な検証が必要です。また、提供されるAI処理の種類やSQL拡張の詳細、パイプラインのパフォーマンスやスケーラビリティに関する制限事項については、公式ドキュメントやサポート情報を参照する必要があります。

### 4. Generally Available: Ingest OTLP signals into Azure Monitor with the OpenTelemetry Collector

**公開日時**: 2026年06月04日 19:45:40 UTC
**リンク**: [Generally Available: Ingest OTLP signals into Azure Monitor with the OpenTelemetry Collector](https://azure.microsoft.com/updates?id=565090)

**アップデートID**: 565090
**情報源**: Azure Updates API

**カテゴリ**: Launched, DevOps, Management and governance, Azure Monitor, Microsoft Build, Open Source

**要約**:

【何が更新されたか】  
Azure MonitorがOpenTelemetry Protocol（OTLP）シグナルのネイティブ取り込みに正式対応しました。

【主な変更点や新機能】  
OpenTelemetry Collectorを利用して、OpenTelemetryでインストルメントされたアプリケーションやプラットフォームからOTLP形式のテレメトリデータを直接Azure Monitorに送信できるようになりました。これにより、分散トレーシングやメトリクス、ログの統合監視がより簡単に実現できます。

【影響を受ける対象】  
OpenTelemetryを導入しているアプリケーションやサービスをAzure Monitorで監視している技術者や開発者が対象です。特にマイクロサービスやクラウドネイティブな環境での監視運用に関わる方にとって重要なアップデートです。

【注意点】  
OTLPシグナルの取り込みにはOpenTelemetry Collectorの設定が必要です。既存の監視構成やデータフローに影響が出る可能性があるため、導入前に十分な検証を行うことを推奨します。

**詳細**:

Azure MonitorにおけるOpenTelemetry Protocol（OTLP）シグナルのネイティブ取り込み機能が一般提供（GA）となりました。本アップデートの背景には、クラウドネイティブアプリケーションの監視やトレーシングの標準化が進む中、OpenTelemetryの普及と、それに伴う多様なテレメトリデータの一元的な管理ニーズの高まりがあります。これまでAzure Monitorへのテレメトリデータ送信には独自のSDKやエージェントが必要でしたが、今回のアップデートにより、OpenTelemetryでインスツルメントされたアプリケーションやプラットフォームから、OTLP形式のデータを直接Azure Monitorへ送信できるようになりました。

具体的には、OpenTelemetry Collectorを利用して、OTLPシグナル（トレース、メトリック、ログなど）をAzure Monitorに取り込むことが可能となります。これにより、OpenTelemetryをサポートする多様な言語やフレームワークで生成されたテレメトリデータを、追加の変換や中継を行うことなく、シームレスにAzure Monitorへ集約できます。技術的な実装方法としては、OpenTelemetry CollectorのExporterとしてAzure Monitorを指定し、OTLPエンドポイントへの送信設定を行います。Collectorは、アプリケーションから受信したOTLPデータをAzure Monitorの受信APIに転送します。

この機能の活用シナリオとしては、マイクロサービスアーキテクチャを採用した分散アプリケーションのパフォーマンス監視や障害解析、複数クラウドやオンプレミス環境にまたがるシステムの統合的な可 observabilityの実現などが挙げられます。たとえば、Kubernetesクラスター上の各種サービスからOTLP形式でトレースやメトリックを収集し、Azure Monitorで一元的に可視化・分析することが可能です。

注意点としては、OpenTelemetry CollectorおよびAzure Monitor側でサポートされるOTLPシグナルの種類やバージョン、データスキーマの互換性に留意する必要があります。また、Collectorの設定やAzure Monitorの受信エンドポイントの認証・権限管理も適切に行う必要があります。

本機能は、Azure Monitorの既存のログ分析やアラート、ダッシュボード機能と連携して活用できます。これにより、OpenTelemetryを活用したモダンな監視基盤をAzure上で効率的に構築・運用することが可能となります。詳細な情報や設定手順については、公式ドキュメントやアップデートページを参照してください。
