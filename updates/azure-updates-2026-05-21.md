**詳細**:

今回のアップデートは、Azure Database for PostgreSQLを利用するアプリケーションにおいて、Microsoft Entra ID（旧称Azure Active Directory）トークンのリフレッシュ機能がPython、.NET、JavaScriptのクライアントライブラリでサポートされたことを示しています。背景としては、クラウドネイティブなアプリケーションの認証管理を簡素化し、セキュリティを高めるために、従来の静的な認証情報（ユーザー名・パスワード等）から、より動的かつ安全なトークンベース認証への移行が進められてきました。今回のアップデートは、その流れの中で、トークンの自動リフレッシュ対応を拡充するものです。

具体的な機能としては、Python、.NET、JavaScriptの各クライアントライブラリが、Microsoft Entra IDによるアクセストークンの有効期限切れ時に自動的にトークンをリフレッシュすることが可能となりました。これにより、アプリケーション側でトークンの有効期限管理や再取得処理を実装する必要がなくなり、認証処理の簡素化と安定運用が実現できます。技術的な仕組みとしては、クライアントライブラリがEntra IDから取得したアクセストークンを保持し、必要に応じてリフレッシュトークンを用いて新しいアクセストークンを取得する動作を自動化しています。これにより、認証セッションの維持が容易になり、ユーザーやサービスアカウントの認証情報漏洩リスクも低減されます。

活用シナリオとしては、Azure Database for PostgreSQLをバックエンドに持つWebアプリケーションやAPIサービス、バッチ処理などで、Microsoft Entra IDによる認証を導入している場合に特に有効です。例えば、企業内の認証基盤としてEntra IDを利用している場合、アプリケーションの認証処理を統一し、管理負荷を軽減することができます。また、複数のAzureサービスと連携する場合でも、同一の認証基盤を用いることでセキュアなアクセス制御が可能となります。

注意点としては、トークンリフレッシュ機能を利用するには、各クライアントライブラリの最新バージョンを使用する必要があります。また、Microsoft Entra IDの設定や権限管理が適切に行われていない場合、トークン取得やリフレッシュ処理が失敗する可能性があります。さらに、アプリケーションの認証フローにおいて、トークンの有効期限やリフレッシュポリシーに関するAzure側の制約を十分に理解しておくことが重要です。

関連するAzureサービスとしては、Microsoft Entra IDを中心に、Azure Database for PostgreSQL、Azure App Service、Azure Functionsなどのサービスと連携する際に、今回のトークンリフレッシュ機能が活用されます。これにより、Azure全体で統一された認証基盤を構築し、運用の効率化とセキュリティ向上を図ることができます。

### 4. Generally Available: Azure Storage Mover Blob-to-Blob migration

**公開日時**: 2026年05月20日 17:00:39 UTC
**リンク**: [Generally Available: Azure Storage Mover Blob-to-Blob migration](https://azure.microsoft.com/updates?id=562753)

**アップデートID**: 562753
**情報源**: Azure Updates API

**カテゴリ**: Launched, Migration, Storage, Azure Storage Mover, Features, Management, Services

**要約**:

【何が更新されたか】  
Azure Storage MoverがBlobコンテナ間のデータ移行（Blob-to-Blob migration）をサポートし、一般提供（GA）されました。

【主な変更点や新機能】  
これまでファイル移行が中心だったStorage Moverですが、今回のアップデートにより、Blobコンテナから別のBlobコンテナへのデータ転送が可能になりました。これにより、リージョン、サブスクリプション、アカウントを跨いだ移行が、完全にマネージドな形で実施できます。

【影響を受ける対象】  
Azure Blob Storageを利用しているユーザーや、データ移行を検討している技術者が対象です。特に、複数リージョンや異なるアカウント間でのデータ移行ニーズがある場合に有用です。

【注意点】  
移行先・移行元のBlobコンテナの設定やアクセス権限に注意が必要です。また、移行作業中のデータ整合性やパフォーマンスへの影響を考慮し、事前にテストを行うことを推奨します。

**詳細**:

本アップデートは、「Azure Storage Mover」がBlobコンテナー間のデータ転送、すなわちBlob-to-Blobマイグレーションを正式にサポートしたことを示しています。これにより、ユーザーは異なるリージョン、サブスクリプション、アカウント間でのデータ移行を、Azureがフルマネージドで提供するエクスペリエンスとして実現できるようになりました。

この機能追加の背景には、企業や組織がクラウドストレージを利用する中で、データの移動や統合、バックアップ、ディザスタリカバリなどの用途で、異なるBlobコンテナー間のシームレスなデータ移行ニーズが高まっていることがあります。従来は、AzCopyや自作のスクリプト、またはサードパーティ製ツールなどを用いてBlobストレージ間のデータ移行を行う必要がありましたが、今回のアップデートにより、Azure Storage Moverを活用したより簡便かつ信頼性の高い移行が可能となりました。

具体的な機能としては、Azure Storage MoverがBlobコンテナーから別のBlobコンテナーへのデータ転送をサポートします。これにより、ユーザーはAzureポータルやAPIを通じて、転送元と転送先のBlobコンテナーを指定し、移行ジョブを作成・管理できます。Azure Storage Moverはフルマネージドサービスであるため、インフラの構築やメンテナンスは不要で、スケーラブルかつ高可用性なデータ転送が実現できます。

技術的な仕組みとしては、Azure Storage MoverはAzureの管理プレーンを通じてジョブの作成・監視を行い、データプレーンではBlobストレージのREST APIを用いてデータのコピーを実行します。これにより、転送中のデータ整合性やエラーリカバリもAzure側で管理されます。

活用シナリオとしては、リージョン間のデータ移行による災害対策、サブスクリプション統合時のデータ集約、アカウント間のデータ移設、ストレージのリファクタリングや最適化などが挙げられます。特に、組織再編やコスト最適化のためにストレージアカウントの再配置が必要な場合に有効です。

注意点としては、サポートされるBlobの種類や転送速度、転送中のデータ整合性、アクセス制御の継承など、詳細な制限事項については公式ドキュメントを参照する必要があります。また、転送元・転送先のストレージアカウントの権限設定やネットワーク構成にも留意が必要です。

関連するAzureサービスとしては、Azure Blob Storage、Azure Storage Account、Azure Portal、Azure Monitorなどが挙げられます。これらと連携することで、移行ジョブの監視や自動化、運用管理が容易になります。

以上のように、本アップデートによりAzure Storage Moverを活用したBlob-to-Blobマイグレーションが正式にサポートされ、Azure環境におけるデータ移行の効率化と運用負荷の軽減が期待できます。
